I'm continuing onBLAST — a community-driven business accountability platform.
Read `app-idea.md` in the repo root before making architectural decisions. Any
"decided" sections in that spec are settled unless the user explicitly changes
them.

Repo / stack:
  - Monorepo with npm workspaces
  - `apps/api`: Fastify + Prisma + zod
  - `apps/web`: Next.js 15 App Router
  - `packages/crypto`: Ed25519 auth + client-side encrypted recovery

Critical constraints — do NOT violate:
  - "Nothing to subpoena": no email, phone, IP, user-agent, device fingerprint.
  - Opinion moderation is community-driven only. Mods only act on hard limits.
  - CSAM/NCMEC is a stub — do NOT mark anything SUBMITTED without real integration.
  - Spec defers: credibility scoring, business reply accounts, formal appeals,
    rate limiting, admin auth for legal intake, job queue, CDN, vote tuning.

Auth model:
  - POST /auth/challenge → nonce
  - POST /auth/verify → Ed25519 sig check, 15m JWT, create account on first sight
  - Recovery: optional, client-encrypted; server stores only ciphertext

What's been built (cumulative):

  API routes:
    GET/POST /auth/challenge, /auth/verify, /auth/recovery/*
    GET      /businesses, /businesses/:slug
    POST     /businesses (auth required)
    GET      /posts  (cursor pagination, authorId filter, returns {posts, nextCursor})
    GET      /posts/:id
    POST     /posts (auth required)
    POST     /votes (auth required; updates author reputationScore)
    GET/POST /petitions
    GET      /petitions/:id (with cosigners + voteSummary)
    POST     /petitions/:id/cosign, /petitions/:id/vote (auth required)
    GET      /moderation-log (cursor paginated)
    GET/POST /legal-demands (POST requires x-operator-key header if OPERATOR_KEY env set)
    GET      /accounts/:id
    GET      /health
    POST     /csam (stub)

  Web pages:
    /                   Home: feed + post composer + business browser (search) +
                        petition list + mod log sidebar
    /auth               Keypair create/recover flow
    /businesses/[slug]  Business profile + inline post composer + paginated posts
    /posts/[id]         Post detail + interactive voting
    /petitions/[id]     Petition detail + Cosign/Vote buttons (status-aware)
    /moderation-log     Paginated public enforcement log, expandable JSON details
    /legal-demands      Lumen-style demand registry + operator intake form
    /accounts/[id]      Account profile: pubkey, rep, counts, paginated post history

  Global:
    - SiteNav sticky bar (logo, Mod log, Legal links, auth state)
    - Global not-found.tsx and error.tsx
    - Seed script: prisma/seed.ts — run with npm.cmd run db:seed

Known blockers:
  - DB not migrated/seeded. Run:
      npm.cmd run db:migrate
      npm.cmd run db:generate  (if needed after migrate)
      npm.cmd run db:seed
  - PowerShell blocks npm.ps1; use npm.cmd.
  - OPERATOR_KEY env var: if set, POST /legal-demands requires
    x-operator-key header. Unset in dev = no check.
  - JWT is 15m with no silent refresh yet — users get auth errors after
    timeout until they re-sign via /auth.

High-value next options:
  1. JWT silent refresh — re-challenge + re-sign automatically when token
     expires. Needs a refresh helper in storage.ts called before each
     authenticated API call.
  2. Author links in business shell + post detail shell — post.author.id
     is available; add <Link href={`/accounts/${post.author.id}`}> to
     businesses/[slug]/shell.tsx postMeta and posts/[id]/shell.tsx meta.
     Home shell already has this link.
  3. Petition target account link — on /petitions/[id], the targetId is
     an account ID; link it to /accounts/[targetId].
  4. Error.tsx per route — only global error.tsx exists; add per-route
     error.tsx for /posts/[id], /petitions/[id], /businesses/[slug] for
     more targeted recovery messaging.
  5. Search bar for petitions — home sidebar petitions panel shows all
     petitions; add a status filter (OPEN / AT_VOTE / PASSED / FAILED).
  6. Deployment config — Dockerfile, .env.example, and a deploy checklist
     (DATABASE_URL, JWT_SECRET, OPERATOR_KEY, WEB_ORIGIN, API_PORT).

Open decisions (do not decide alone):

  2. localStorage → IndexedDB/WebCrypto for key storage (v2)
  3. Vote-weight curve tuning and petition thresholds
  4. JWT TTL strategy (extend vs. silent re-sign)
