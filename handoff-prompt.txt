I'm continuing onBLAST — a community-driven business accountability platform.
Read `app-idea.md` in the repo root before making architectural decisions. Any
"decided" sections in that spec are settled unless the user explicitly changes
them.

Repo / stack:
  - Monorepo with npm workspaces
  - `apps/api`: Fastify + Prisma + zod
  - `apps/web`: Next.js 15 App Router
  - `packages/crypto`: Ed25519 auth + client-side encrypted recovery

Critical constraints — do NOT violate:
  - "Nothing to subpoena": do not add email, phone, IP, user-agent, device
    fingerprint, or similar identifiers to schema, logs, or convenience
    instrumentation.
  - Moderation of opinion content stays community-driven. Mods only act on
    hard-limit categories.
  - CSAM/NCMEC integration is still a stub. Do NOT mark anything `SUBMITTED`
    without a real integration.
  - The spec intentionally defers: credibility scoring, business reply
    accounts, formal appeals, rate limiting, admin auth for legal intake, job
    queue, CDN, and final tuning for vote/petition parameters.

Auth model:
  - `POST /auth/challenge` returns a nonce
  - `POST /auth/verify` verifies Ed25519 signature, issues 15m JWT, and creates
    the account on first sight
  - Recovery is optional and client-encrypted; the server only stores
    ciphertext

Known blockers / caveats:
  - DB not migrated or seeded. Run:
      npm.cmd run db:migrate
      npm.cmd run db:generate  (if needed)
      npm.cmd run db:seed
  - PowerShell execution policy blocks npm.ps1; use npm.cmd.
  - /legal-demands intake form is currently unauthenticated. Before
    exposing publicly, add operator-only auth to POST /legal-demands in
    apps/api/src/modules/legal-demands/routes.ts.

High-value next implementation options:
  1. **Operator auth for legal intake** — add a simple shared-secret or
     separate JWT-based guard to POST /legal-demands so only the designated
     agent can log demands. The GET remains public.
  2. **Feed pagination** — GET /posts currently returns top 50. Add a
     cursor-based "Load more" to the home shell and business shell, matching
     the pattern already used in the mod log page.
  3. **Post linking on petition detail** — the petition's targetId is an
     account, but posts by that account could be linked. A "View posts by
     this account" query (GET /posts?authorId=...) would complete the
     investigation flow. Needs a new query param on the posts route.
  4. **Account profile page** — /accounts/[id] showing reputation, post
     history, petition history. Useful for the community to evaluate credibility.
     Requires GET /accounts/:id on the API.
  5. **JWT refresh / session persistence** — the access token is 15 min.
     There's no silent refresh yet. A user who stays on the page past 15 min
     will get auth errors until they re-sign via /auth. Add a refresh flow
     (either re-sign on challenge automatically, or extend JWT TTL in config).
  6. **Error boundary pages** — Next.js error.tsx / not-found.tsx stubs for
     /posts/[id], /petitions/[id], and /businesses/[slug] so 404s and API
     errors show a readable page instead of a crash.
